## Context

Longgu has two valid product shapes:

- standalone CLI/app, where Longgu owns provider calls and model routing;
- AI-assistant harness, where Longgu prepares context and stores artifacts while the host assistant supplies generation.

Issue #4 asks to avoid forcing the first shape onto Claude Code users.

## Decision

Add an explicit host-LLM drafting workflow instead of trying to access Claude Code internals:

- `write chapter --host-prompt --id <id>` builds the same context pack used by provider drafting, renders the chapter prompt, and writes `host-prompts/<id>.prompt.md`.
- `write chapter --input <path> --id <id>` reads a Markdown draft generated by the host assistant, normalizes the title the same way provider output is normalized, writes `chapters/<id>.md`, and creates a run record.
- Provider-backed `write chapter --id <id>` continues to use routing and OpenAI-compatible generation.

## Config Boundary

`provider` becomes optional at the schema level. Commands that require provider access use an explicit guard. This allows `context build`, `write --host-prompt`, and `write --input` to run in host-only workspaces while preserving clear failures for provider-backed commands.

## Run Records

Host import run records use:

- `provider: "host-llm"`
- `model: "host-llm"`
- `modelProfile: "host"`
- `estimatedCost: 0`
- normal `prompt.md`, `context.json`, `output.md`, and `metadata.json`

This keeps cost reporting and run inspection simple without hiding that the generation happened outside Longgu's provider adapter.

## Tradeoffs

- This does not automate direct calls into Claude Code's model. That is intentional: there is no stable CLI API for Longgu to invoke the host conversation context.
- The workflow is two-step in host mode, but it is explicit, reviewable, and immediately useful.
- Existing provider-backed generation remains the right path for unattended runs and standalone use.
